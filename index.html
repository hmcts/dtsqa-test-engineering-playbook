    • Best Practices
      • Manual Testing
          • Any test case that is not suitable for automation testing should be included in manual test pack
          • Record all your manual test cases in Zephyr or similar
          • Make sure to include all necessary elements when you write your test case
          • Test Data Preparation
          • Don’t forget to execute your manual tests for every release
      • Integration Testing
          • Write Integration Tests wherever you serialize/deserialize data
          • Don’t test business logic with integration tests
          • Don’t rely on existing data
          • Don’t use hard-coded values
          • Use Test Doubles
          • Log extensively in integration tests
          • Keep Unit tests and Integration tests separate
          • Don’t mark tests as Ignored
          • Run integration tests as part of the CI/CD pipeline
      • Exploratory Testing
          • Add Exploratory testing to your existing test Strategy
          • Structure your exploratory testing
          • Use a destructive mindset
          • Document everything you find for later
          • Try and automate the bugs you found through Exploratory Testing
          • Exploratory testing is not a good idea for regulated or compliance based applications
      • End to End (Functional) Testing
          • Leverage the test Automation
          • Reconsider failing and flaky test
          • Avoid Exception Testing
          • Use REST API End to End tests
          • Move the tests to Lower Levels of the Pyramid
          • E2E Tests in the pipeline
          • Don’t ignore flaky tests. React as soon as possible.
          • E2E tests should execute later in the build pipeline
          • Create test data smartly
        • Specific to UI
          • Write as few end to end tests as possible
          • Think before you use Xpaths
          • Use Conditional Waits
          • Ensure that e2e Tests failures in the pipeline are recreatable locally
      • Unit Testing
          • Tests Should Be Fast
          • Tests should be reliable
          • Tests Should Be Simple and Readable
          • Arrange, Act, Assert
          • Name your tests well
          • Stick to single assertion per test
          • Tests should be deterministic
          • Test Shouldn’t Duplicate Implementation Logic
          • Tests should be independent
          • Use Test Doubles
          • Don’t mark tests as Ignored
          • Never write unit tests after the production code
          • Unit tests should run as part of the CI/CD Pipeline
          • Consider Data Driven with your unit testing framework
          • Consider using Testing techniques (Equivalence class Partition, Boundary value Analysis, State Transition Table)
      • Contract Testing
          • Terminology
          • What is Pact good for?​
          • What is it not good for?​
          • Write Consumer Contract tests for all API interactions
          • Follow the naming standards for participants
          • Avoid random data in expectations
          • Match Types or Regexes instead of hard-coded static data
          • One Contract test per Consumer-Provider interaction
          • Publish the Consumer Contract to the PactBroker
          • Configure your provider build to fetch the pact from the broker
          • Use the can-i-deploy tool in your pipeline to check whether its safe to deploy
          • Don’t mark tests as Ignored
      • Testing Frontend Applications
          • Write as few end to end tests as possible
          • Create test data smartly
          • Think before you use Xpaths
          • Use Conditional Waits
      • Testing on Power Platform
          • Identify and test patterns and outcomes of your flows
          • Automated end to end tests are very important
          • Understand the business process
          • Shift Testing to Left
